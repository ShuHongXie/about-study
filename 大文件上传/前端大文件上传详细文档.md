# 前端大文件上传详细文档

# 一、文档概述

## 1.1 文档目的

本文档详细阐述前端大文件上传的核心方案、实现流程、代码示例、常见问题及优化方案，适用于前端开发人员进行大文件上传功能开发、调试及面试参考，确保开发过程规范、高效，解决大文件上传超时、易失败、体验差等核心痛点。

## 1.2 适用场景

适用于需要上传超大文件（通常指100MB及以上，如视频、压缩包、大型文档等）的前端场景，例如：视频上传平台、云存储文件上传、大型项目附件上传等，可直接参考本文档进行方案设计和代码落地。

## 1.3 核心痛点（解决目标）

- 单次上传大文件，请求超时、服务器拒收（超出请求大小限制）；

- 网络中断、页面刷新后，需重新上传全量文件，耗时久、体验差；

- 重复上传相同文件，浪费带宽和服务器存储空间；

- 上传过程中页面卡顿、无响应，无法正常交互；

- 分片上传后，后端合并失败导致文件损坏；

- 上传进度无法实时展示，用户体验差。

# 二、核心技术方案

前端大文件上传的核心方案为：**文件分片 + MD5唯一标识 + Web Worker + 并发上传 + 断点续传 + 秒传 + 失败重试 + 进度实时计算**，各技术点协同作用，解决上述所有核心痛点，方案适配主流浏览器（Chrome、Firefox、Edge等），兼容Vue、React等主流前端框架。

## 2.1 核心技术点说明

| 技术点       | 核心作用                                   | 实现方式                                          |
| ------------ | ------------------------------------------ | ------------------------------------------------- |
| 文件分片     | 化整为零，降低单次上传压力，避免超时       | 使用File.slice()切割文件，生成固定大小分片        |
| MD5唯一标识  | 文件唯一指纹，实现秒传、断点续传、合并校验 | 通过SparkMD5计算文件二进制数据的MD5值             |
| Web Worker   | 隔离主线程，避免MD5计算阻塞页面交互/渲染   | 创建独立Worker线程，异步计算MD5和处理分片         |
| 并发上传     | 提升上传速度，避免单分片上传耗时过久       | 用并发池控制上传数量（3~6个），批量上传分片       |
| 断点续传     | 中断后无需重传全量，仅上传缺失分片         | 前端预检+后端记录已上传分片，过滤已传内容         |
| 秒传         | 重复文件无需上传，直接返回成功             | MD5校验，后端判断文件是否已存在                   |
| 失败重试     | 单个分片上传失败，避免整个文件重新上传     | 单个分片失败后自动重试（最多3次），仍失败提示用户 |
| 进度实时计算 | 展示上传进度，提升用户体验                 | 根据已上传分片大小/数量，计算总进度并实时更新     |

## 2.2 方案优势

- 稳定性：分片上传降低单次请求压力，失败重试、断点续传保证上传不中断；

- 高效性：并发上传、秒传大幅提升上传速度，节省带宽和时间；

- 流畅性：Web Worker隔离主线程，避免页面卡顿，进度实时展示提升体验；

- 兼容性：适配主流浏览器和前端框架，无需额外依赖复杂插件；

- 可扩展性：支持分片大小、并发数自定义，可适配不同服务器配置。

# 三、完整实现流程（前端+后端协同）

本文档以“1GB视频文件上传”为例，分片大小设为5MB，并发数设为3，完整流程分为8个步骤，前端负责切片、计算MD5、上传分片，后端负责校验、存储分片、合并文件，协同完成大文件上传。

## 3.1 步骤1：前端初始化（接收文件+配置参数）

### 3.1.1 功能说明

用户通过文件选择器选择大文件后，前端获取文件基础信息（名称、大小、类型），配置分片大小、并发数等核心参数，为后续切片、上传做准备。

### 3.1.2 核心代码（通用版/React示例）

```javascript
// React示例：文件选择事件处理
import { useState } from 'react';

  // 存储文件信息、上传进度、MD5等状态

  };

  return (
```

## 3.2 步骤2：前端文件切片（生成fileChunkList）

### 3.2.1 功能说明

利用浏览器原生API `File.slice()`，将大文件切割成固定大小的分片（最后一片可能小于分片大小），生成分片列表`fileChunkList`，每个分片包含分片文件、分片索引、分片大小等信息，用于后续MD5计算和分片上传。

### 3.2.2 核心代码

```javascript
/**
 * 生成文件切片
 * @param {File} file - 原始大文件
 * @param {number} chunkSize - 分片大小（单位：字节）
 * @returns {Array} - 文件分片列表
 */
function createFileChunk(file, chunkSize) {
  const fileChunkList = [];
  let cur = 0; // 当前切割位置
  while (cur < file.size) {
    // 切割分片：从cur位置开始，切割chunkSize大小的内容
    const chunkFile = file.slice(cur, cur + chunkSize);
    fileChunkList.push({
      chunkFile: chunkFile, // 分片文件（Blob类型）
      chunkIndex: cur / chunkSize, // 分片索引（从0开始，用于后端合并排序）
      chunkSize: Math.min(chunkSize, file.size - cur), // 当前分片大小（最后一片可能更小）
      progress: 0, // 单个分片上传进度（用于实时更新）
    });
    cur += chunkSize;
  }
  return fileChunkList;
}

// 调用示例（在开始上传时执行）
const fileChunkList = createFileChunk(file, CONFIG.CHUNK_SIZE);
console.log("文件切片完成，分片数量：", fileChunkList.length);
```

### 3.2.3 注意事项

- 分片大小建议设置为2~10MB：过小会导致分片数量过多、请求次数过多，增加服务器压力；过大则失去分片意义，仍可能超时；

- `File.slice()` 切割的是Blob对象，不改变原始文件，仅生成新的Blob分片；

- 分片索引必须连续且唯一，用于后端合并时按顺序拼接分片。

## 3.3 步骤3：前端计算MD5（Web Worker异步计算）

### 3.3.1 功能说明

通过Web Worker创建独立线程，异步计算整个文件的MD5值（fileHash），作为文件的唯一标识。避免在主线程计算MD5（CPU密集型任务）导致页面卡顿、无响应，同时通过Web Worker实时返回计算进度。

### 3.3.2 依赖准备

使用前端MD5计算库`spark-md5`（轻量、高效、适配Web Worker），安装方式：

```bash
npm install spark-md5 --save
# 或 yarn add spark-md5

```

### 3.3.3 核心代码（Web Worker文件：hashWorker.js）

```javascript
// 引入spark-md5（需确保路径正确）
import SparkMD5 from "./spark-md5.min.js";

// 生成文件切片（和主线程方法一致，避免主线程传递大量分片数据）
function createFileChunk(file, chunkSize) {
  let fileChunkList = [];
  let cur = 0;
  while (cur < file.size) {
    fileChunkList.push({ chunkFile: file.slice(cur, cur + chunkSize) });
    cur += chunkSize;
  }
  return fileChunkList;
}

// 异步计算所有分片的MD5（递归处理，避免一次性加载过多分片）
async function calculateChunksHash(fileChunkList) {
  const spark = new SparkMD5.ArrayBuffer(); // 初始化MD5计算器
  let percentage = 0; // MD5计算进度
  let count = 0; // 已处理的分片数量

  // 递归处理每个分片，读取二进制数据并追加到MD5计算器
  async function loadNext(index) {
    if (index >= fileChunkList.length) {
      return spark.end(); // 所有分片处理完成，返回最终MD5值
    }

    return new Promise((resolve, reject) => {
      const reader = new FileReader();
      // 读取当前分片为ArrayBuffer（唯一能用于MD5计算的格式）
      reader.readAsArrayBuffer(fileChunkList[index].chunkFile);
      // 读取成功
      reader.onload = (e) => {
        count++;
        spark.append(e.target.result); // 将当前分片的二进制数据追加到MD5计算中
        // 更新计算进度，发送到主线程
        percentage = Math.round((count / fileChunkList.length) * 100);
        self.postMessage({ type: "hashProgress", data: percentage });
        resolve(loadNext(index + 1)); // 递归处理下一个分片
      };
      // 读取失败
      reader.onerror = (err) => {
        reject(err);
      };
    });
  }

  try {
    // 从第0个分片开始计算，获取最终MD5值
    const fileHash = await loadNext(0);
    // 将MD5值、文件分片列表发送到主线程
    self.postMessage({
      type: "hashSuccess",
      data: { fileHash, fileChunkList },
    });
    self.close(); // 关闭Worker线程，释放资源
  } catch (err) {
    // 计算失败，发送错误信息到主线程
    self.postMessage({ type: "hashError", data: err.message });
    self.close();
  }
}

// 监听主线程发送的消息（接收文件和分片大小）
self.addEventListener(
  "message",
  async (e) => {
    try {
      const { file, chunkSize } = e.data;
      const fileChunkList = createFileChunk(file, chunkSize); // 生成文件切片
      await calculateChunksHash(fileChunkList); // 开始计算MD5
    } catch (err) {
      self.postMessage({ type: "hashError", data: err.message });
      self.close();
    }
  },
  false,
);

// 监听Worker线程错误，发送错误信息到主线程
self.addEventListener("error", (event) => {
  self.postMessage({ type: "hashError", data: event.message });
  self.close();
});
```

### 3.3.4 核心代码（主线程调用Web Worker）

```javascript
/**
 * 调用Web Worker计算文件MD5
 * @param {File} file - 原始大文件
 * @param {number} chunkSize - 分片大小
 * @returns {Promise} - 返回fileHash和fileChunkList
 */
function calculateFileHash(file, chunkSize) {
  return new Promise((resolve, reject) => {
    // 创建Web Worker实例（路径需和实际文件一致）
    const worker = new Worker(new URL("./hashWorker.js", import.meta.url));

    // 发送文件和分片大小到Worker线程
    worker.postMessage({ file, chunkSize });

    // 监听Worker线程发送的消息（进度、结果、错误）
    worker.addEventListener("message", (e) => {
      const { type, data } = e.data;
      switch (type) {
        case "hashProgress":
          // 更新MD5计算进度（可选，用于展示）
          setProgress(data);
          break;
        case "hashSuccess":
          // MD5计算成功，返回结果
          resolve(data); // data = { fileHash, fileChunkList }
          break;
        case "hashError":
          // MD5计算失败
          reject(new Error(data));
          break;
      }
    });

    // 监听Worker线程错误
    worker.addEventListener("error", (err) => {
      reject(new Error(`MD5计算失败：${err.message}`));
      worker.terminate(); // 终止Worker线程
    });
  });
}

// 调用示例（在开始上传时执行）
const { fileHash, fileChunkList } = await calculateFileHash(
  file,
  CONFIG.CHUNK_SIZE,
);
setFileHash(fileHash);
console.log("MD5计算完成，文件唯一标识：", fileHash);
```

### 3.3.5 关键说明

- 为什么要转ArrayBuffer？：MD5算法（SparkMD5）仅支持二进制底层的ArrayBuffer格式，File/Blob是高层封装对象，需通过FileReader.readAsArrayBuffer()转成ArrayBuffer才能获取原始二进制数据，保证MD5计算准确；

- 为什么用Web Worker？：大文件MD5计算是CPU密集型长任务，若在主线程执行，会阻塞页面交互和渲染，导致页面卡顿、无响应，Web Worker隔离主线程，保证页面流畅；

- MD5唯一性：同一文件（内容不变），无论文件名、后缀如何修改，MD5值始终不变；不同文件（内容不同），MD5值碰撞概率极低，可忽略不计，完全满足业务需求。

## 3.4 步骤4：前端预检（秒传/断点续传校验）

### 3.4.1 功能说明

MD5计算完成后，前端向后端发送预检请求，携带文件唯一标识fileHash，后端校验该文件是否已存在、已上传的分片列表，前端根据校验结果，决定执行秒传、断点续传或全新上传，最大程度减少无效上传。

### 3.4.2 核心代码（前端）

```javascript
/**
 * 预检请求：查询文件状态（秒传/断点续传）
 * @param {string} fileHash - 文件唯一标识（MD5）
 * @param {File} file - 原始文件（用于获取文件名、大小等信息）
 * @returns {Promise} - 后端返回的文件状态
 */
async function checkFileStatus(fileHash, file) {
  try {
    const res = await axios.post(CONFIG.UPLOAD_URL.checkFile, {
      fileHash: fileHash, // 文件MD5
      fileName: file.name, // 文件名（用于后端存储时命名）
      fileSize: file.size, // 文件总大小（用于校验）
      fileType: file.type, // 文件类型（用于后端校验）
    });
    // 后端返回格式（需和后端协商，示例如下）
    // { isExist: false, uploadedChunks: [0,1,2], message: '文件未存在，可上传' }
    return res.data;
  } catch (err) {
    throw new Error(`预检请求失败：${err.message}`);
  }
}

// 调用示例
const checkResult = await checkFileStatus(fileHash, file);
const { isExist, uploadedChunks } = checkResult;

// 处理预检结果
if (isExist) {
  // 秒传：文件已存在，直接提示上传成功
  setProgress(100);
  setUploading(false);
  alert("文件已存在，秒传成功！");
  return;
}
console.log("已上传分片索引：", uploadedChunks);
```

### 3.4.3 后端核心逻辑（简要说明）

- 接收前端传递的fileHash，查询数据库/文件存储目录，判断该fileHash对应的完整文件是否存在；

- 若存在，返回`isExist: true`，前端执行秒传；

- 若不存在，查询该fileHash对应的已上传分片索引列表（uploadedChunks），返回给前端，用于前端过滤已传分片；

- 后端需将“fileHash-已上传分片”的关联关系存储到数据库/缓存（如Redis），便于后续断点续传校验。

## 3.5 步骤5：前端过滤分片（仅上传缺失分片）

### 3.5.1 功能说明

根据后端返回的已上传分片列表（uploadedChunks），过滤掉fileChunkList中已上传的分片，只保留未上传的分片（needUploadChunks），避免重复上传，实现断点续传。

### 3.5.2 核心代码

```javascript
/**
 * 过滤已上传的分片，只保留未上传的分片
 * @param {Array} fileChunkList - 完整的文件分片列表
 * @param {Array} uploadedChunks - 后端返回的已上传分片索引列表
 * @returns {Array} - 需要上传的分片列表
 */
function filterUploadedChunks(fileChunkList, uploadedChunks) {
  if (!uploadedChunks || uploadedChunks.length === 0) {
    return fileChunkList; // 无已上传分片，返回全部
  }
  // 过滤：保留分片索引不在uploadedChunks中的分片
  return fileChunkList.filter(
    (chunk) => !uploadedChunks.includes(chunk.chunkIndex),
  );
}

// 调用示例
const needUploadChunks = filterUploadedChunks(fileChunkList, uploadedChunks);
console.log("需要上传的分片数量：", needUploadChunks.length);

if (needUploadChunks.length === 0) {
  // 无需要上传的分片，直接触发合并
  await mergeChunks(fileHash, file, fileChunkList.length);
  return;
}
```

## 3.6 步骤6：前端并发上传分片（失败重试）

### 3.6.1 功能说明

使用并发池控制分片上传数量（避免请求过多导致浏览器/服务器阻塞），逐个上传needUploadChunks中的分片，每个分片携带fileHash、chunkIndex等信息，便于后端关联分片；单个分片上传失败后，自动重试（最多3次），仍失败则提示用户暂停上传。

### 3.6.2 核心代码

```javascript
/**
 * 单个分片上传（带失败重试）
 * @param {Object} chunk - 单个分片信息
 * @param {string} fileHash - 文件MD5
 * @param {number} retryCount - 剩余重试次数
 * @returns {Promise} - 分片上传结果
 */
async function uploadSingleChunk(chunk, fileHash, retryCount) {
  const { chunkFile, chunkIndex } = chunk;
  // 构建FormData（用于上传文件）
  const formData = new FormData();
  formData.append("fileHash", fileHash);
  formData.append("chunkIndex", chunkIndex);
  formData.append("chunkFile", chunkFile); // 分片文件（Blob类型）

  try {
    await axios.post(CONFIG.UPLOAD_URL.uploadChunk, formData, {
      // 实时更新单个分片上传进度
      onUploadProgress: (e) => {
        if (e.lengthComputable) {
          chunk.progress = Math.round((e.loaded / e.total) * 100);
          // 计算总上传进度并更新
          calculateTotalProgress(fileChunkList);
        }
      },
      timeout: 60000, // 超时时间：60秒（避免分片上传超时）
    });
    return true; // 上传成功
  } catch (err) {
    // 上传失败，重试
    if (retryCount > 0) {
      console.log(`分片${chunkIndex}上传失败，剩余重试次数：${retryCount - 1}`);
      return uploadSingleChunk(chunk, fileHash, retryCount - 1);
    } else {
      throw new Error(`分片${chunkIndex}上传失败，已达到最大重试次数`);
    }
  }
}

/**
 * 计算总上传进度
 * @param {Array} fileChunkList - 完整的文件分片列表
 */
function calculateTotalProgress(fileChunkList) {
  const totalSize = fileChunkList.reduce(
    (sum, chunk) => sum + chunk.chunkSize,
    0,
  );
  const uploadedSize = fileChunkList.reduce((sum, chunk) => {
    return sum + (chunk.progress / 100) * chunk.chunkSize;
  }, 0);
  // 更新总进度（保留2位小数）
  const totalProgress = Math.round((uploadedSize / totalSize) * 10000) / 100;
  setProgress(totalProgress);
}

/**
 * 并发上传所有需要上传的分片
 * @param {Array} needUploadChunks - 需要上传的分片列表
 * @param {string} fileHash - 文件MD5
 * @param {number} concurrentNum - 并发上传数
 * @returns {Promise} - 所有分片上传结果
 */
async function uploadChunks(needUploadChunks, fileHash, concurrentNum) {
  // 生成上传Promise列表（每个分片对应一个上传Promise）
  const uploadPromises = needUploadChunks.map((chunk) => {
    return () => uploadSingleChunk(chunk, fileHash, CONFIG.RETRY_COUNT);
  });

  // 并发池控制：每次最多执行concurrentNum个上传请求
  const pool = [];
  for (const upload of uploadPromises) {
    const p = upload();
    pool.push(p);
    // 上传成功后，从并发池中移除
    p.then(() => pool.splice(pool.indexOf(p), 1));
    // 若并发池已满，等待其中一个上传完成后再继续
    if (pool.length >= concurrentNum) {
      await Promise.race(pool);
    }
  }

  // 等待所有分片上传完成
  await Promise.all(pool);
  console.log("所有需要上传的分片均上传完成");
}

// 调用示例
await uploadChunks(needUploadChunks, fileHash, CONFIG.CONCURRENT_NUM);
```

### 3.6.3 后端核心逻辑（简要说明）

- 接收前端传递的fileHash、chunkIndex、chunkFile；

- 创建临时存储目录（如：uploads/{fileHash}/），将分片文件按“chunkIndex”命名存储（如：0、1、2...）；

- 将当前上传的chunkIndex添加到“fileHash-已上传分片”的关联关系中（更新数据库/缓存）；

- 返回上传成功标识，供前端判断是否继续上传。

### 3.6.4 注意事项

- 并发数建议设置为3~6个：过多会导致浏览器请求阻塞、服务器压力过大；过少则上传速度慢；

- 超时时间建议设置为60秒以上：避免网络波动导致分片上传超时；

- 失败重试仅针对单个分片：避免因单个分片失败，导致整个文件重新上传；

- 进度计算：总进度 = 已上传分片大小之和 / 文件总大小，确保进度展示准确。

## 3.7 步骤7：前端触发合并 - 通知后端合并分片

### 3.7.1 功能说明

所有需要上传的分片均上传完成后，前端向后端发送合并请求，携带fileHash、文件名、总分片数等信息，通知后端将该fileHash对应的所有分片，按chunkIndex顺序合并成完整文件。

### 3.7.2 核心代码（前端）

```javascript
/**
 * 通知后端合并分片
 * @param {string} fileHash - 文件MD5
 * @param {File} file - 原始文件（用于获取文件名、类型）
 * @param {number} totalChunks - 总分片数（用于后端校验分片完整性）
 * @returns {Promise} - 合并结果
 */
async function mergeChunks(fileHash, file, totalChunks) {
  try {
    const res = await axios.post(CONFIG.UPLOAD_URL.mergeChunk, {
      fileHash: fileHash,
      fileName: file.name,
      totalChunks: totalChunks,
      fileType: file.type,
      fileSize: file.size,
    });
    // 合并成功，更新状态
    setProgress(100);
    setUploading(false);
    alert(`文件上传成功！文件地址：${res.data.fileUrl}`);
    return res.data;
  } catch (err) {
    setUploading(false);
    throw new Error(`分片合并失败：${err.message}`);
  }
}

// 调用示例（所有分片上传完成后执行）
await mergeChunks(fileHash, file, fileChunkList.length);
```

### 3.7.3 后端核心逻辑（简要说明）

- 接收前端传递的fileHash、totalChunks等信息；

- 校验：查询该fileHash对应的已上传分片数量，是否等于totalChunks（确保分片完整，无缺失）；

- 合并：按chunkIndex顺序（0→1→2...）读取临时目录中的所有分片文件，拼接成完整文件；

- 清理：合并完成后，删除临时分片目录（节省存储空间）；

- 存储：将完整文件移动到最终存储目录，更新数据库（记录fileHash、文件路径、文件名等信息）；

- 返回：返回文件访问地址（fileUrl），供前端展示或使用。

## 3.8 步骤8：上传完成 - 异常处理与状态重置

### 3.8.1 功能说明

上传成功/失败后，更新前端状态（进度、按钮状态），提示用户结果；若上传失败，提供重试按钮，方便用户重新上传；同时重置相关状态，避免影响下次上传。

### 3.8.2 核心代码（异常处理封装）

```javascript
// 完整的上传入口函数（封装所有步骤，统一异常处理）
async function handleStartUpload() {
  if (!file) {
    alert("请先选择文件");
    return;
  }

  setUploading(true);
  try {
    // 步骤2：生成文件切片
    const fileChunkList = createFileChunk(file, CONFIG.CHUNK_SIZE);
    // 步骤3：计算文件MD5
    const { fileHash } = await calculateFileHash(file, CONFIG.CHUNK_SIZE);
    setFileHash(fileHash);
    // 步骤4：预检请求（秒传/断点续传）
    const checkResult = await checkFileStatus(fileHash, file);
    if (checkResult.isExist) {
      setProgress(100);
      setUploading(false);
      alert("文件已存在，秒传成功！");
      return;
    }
    // 步骤5：过滤已上传分片
    const needUploadChunks = filterUploadedChunks(
      fileChunkList,
      checkResult.uploadedChunks,
    );
    if (needUploadChunks.length === 0) {
      // 无需要上传的分片，直接合并
      await mergeChunks(fileHash, file, fileChunkList.length);
      return;
    }
    // 步骤6：并发上传分片
    await uploadChunks(needUploadChunks, fileHash, CONFIG.CONCURRENT_NUM);
    // 步骤7：通知后端合并分片
    await mergeChunks(fileHash, file, fileChunkList.length);
  } catch (err) {
    // 统一捕获所有步骤的错误
    setUploading(false);
    setProgress(0);
    alert(`上传失败：${err.message}，请重试`);
    console.error("上传失败详情：", err);
  }
}
```

# 四、常见问题及解决方案

| 常见问题                | 产生原因                                                                | 解决方案                                                                                                                         |
| ----------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| MD5计算卡顿、页面无响应 | 未使用Web Worker，MD5计算（CPU密集型）阻塞主线程                        | 必须使用Web Worker异步计算MD5，隔离主线程                                                                                        |
| 分片上传超时            | 分片过大、网络波动、超时时间设置过短                                    | 1. 调整分片大小（2~5MB）；2. 延长超时时间（≥60秒）；3. 增加失败重试次数                                                          |
| 秒传失效                | 1. MD5计算错误；2. 后端未正确存储fileHash关联关系；3. 文件内容被修改    | 1. 检查MD5计算逻辑（确保转ArrayBuffer）；2. 后端校验fileHash存储逻辑；3. 提示用户“文件内容已修改，无法秒传”                      |
| 断点续传失效            | 1. 后端未记录已上传分片；2. fileHash计算错误；3. 页面刷新后丢失fileHash | 1. 后端完善已上传分片记录逻辑；2. 校验MD5计算；3. 将fileHash存储到localStorage，刷新后恢复                                       |
| 合并后文件损坏          | 1. 分片缺失；2. 后端合并时顺序错误；3. 分片上传不完整                   | 1. 合并前校验分片数量；2. 后端按chunkIndex顺序合并；3. 前端确保分片上传完成后再触发合并                                          |
| 并发上传时浏览器卡顿    | 并发数设置过多，导致浏览器请求阻塞                                      | 调整并发数为3~6个，避免过多请求同时发起                                                                                          |
| 大文件MD5计算耗时过久   | 文件过大、CPU性能不足                                                   | 1. 优化MD5计算逻辑（递归分片处理，避免一次性加载）；2. 展示MD5计算进度，提升用户体验；3. 可选择“分片MD5+整体MD5”混合计算（可选） |

# 五、优化方案（进阶）

## 5.1 分片大小自适应

根据用户网络速度，动态调整分片大小：网络好（上传速度≥1MB/s），分片大小设为5~10MB；网络差（上传速度<1MB/s），分片大小设为2~3MB，减少分片上传失败概率。

## 5.2 分片MD5校验（可选）

除了计算整个文件的MD5，可额外计算每个分片的MD5，上传分片时携带分片MD5，后端校验分片完整性，避免分片上传过程中数据丢失、损坏。

## 5.3 上传暂停/继续功能

添加暂停按钮，点击暂停时，终止当前所有上传请求（并发池清空）；点击继续时，重新执行预检请求，过滤已上传分片，继续上传未完成的分片。

## 5.4 本地缓存优化

将fileHash、fileChunkList、已上传分片进度等信息存储到localStorage，页面刷新后，自动读取缓存信息，无需重新切片、计算MD5，提升体验。

## 5.5 多文件上传支持

封装上传组件，支持同时选择多个大文件，分别处理每个文件的切片、MD5计算、上传流程，用队列管理多个文件的上传顺序，避免相互干扰。

## 5.6 后端接口优化协同

和后端协商，优化接口响应速度：1. 预检接口、合并接口做缓存处理；2. 分片存储使用分布式存储（如OSS），提升分片上传和读取速度；3. 后端支持分片批量校验、批量合并（可选）。
